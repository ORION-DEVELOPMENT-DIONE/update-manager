# Orion Update Manager

Ansible-based auto-provisioning and update system for Orion devices (Orange Pi gateways). Handles network configuration, VPN access, screen-manager deployment, and scheduled self-updates.

## How It Works

Every Orion Gateway runs `ansible-pull` at midnight via a cron job. This pulls the latest version of this repo from GitHub and executes `local.yml` against the local machine. The result is that every Orion device in the field self-updates every 24 hours with zero manual intervention.

```
┌──────────────────────────────────────────────────────────────────┐
│                     Nightly Update Flow                          │
│                                                                  │
│  00:00 → cron fires ansible-pull                                 │
│       → git clone/pull update-manager from GitHub                │
│       → execute local.yml                                        │
│           ├── pre_tasks: apt update, install deps, load vault    │
│           ├── role: network-config                                │
│           ├── role: tailscale-auth (OAuth key rotation)           │
│           ├── role: tailscale (VPN connection)                    │
│           ├── role: screen-manager (deploy/update app)            │
│           ├── role: auto-update (ensure cron persists)            │
│           └── post_tasks: log completion                          │
└──────────────────────────────────────────────────────────────────┘
```

## Roles

### network-config

Deploys `/etc/network/interfaces` to configure the Orange Pi's network interfaces. NetworkManager handles WiFi; this role sets the baseline interface definitions for loopback and optional wired interfaces.

**Files managed:** `/etc/network/interfaces`
**Tags:** `network`

### tailscale-auth

Manages Tailscale authentication keys via OAuth. Checks if the current key is still valid (based on creation date stored in `/etc/tailscale/key_metadata.json`), and generates a fresh key from the Tailscale API when expired.

**Key features:**
- 30-day key validity (configurable via `tailscale_key_validity_days`)
- Keys are reusable, non-ephemeral, pre-authorized with `tag:orion`
- OAuth credentials sourced from Ansible Vault (`vault_tailscale_oauth_client_id`, `vault_tailscale_oauth_client_secret`)
- Key stored at `/etc/tailscale/auth_key` with mode 0600

**Tags:** `tailscale`
**Condition:** Only runs when `tailscale_oauth_client_id` is defined

### tailscale

Installs Tailscale from the official APT repository, starts the daemon, and connects to the Tailscale network using the auth key generated by `tailscale-auth`.

**Key features:**
- Installs from `pkgs.tailscale.com/stable/ubuntu jammy`
- Hostname set to `orion-<device-hostname>` for easy identification
- Accepts routes and enables SSH access (`--accept-routes --ssh`)
- Handles "already in use" errors gracefully

**Tags:** `tailscale`
**Condition:** Only runs when `tailscale_oauth_client_id` is defined

### screen-manager

Clones or updates the screen-manager repository from GitHub, installs Python dependencies, deploys the systemd service file, and manages the service lifecycle.

**Key features:**
- Clones from `development` branch of `ORION-DEVELOPMENT-DIONE/screen-manager.git`
- Tracks version via `.version` file (git short hash)
- Creates `logs/` directory with correct ownership
- Only restarts the service when the service file itself changes (not on every pull)
- Runs as `root` (required for SPI/GPIO hardware access)

**Tags:** `screen-manager`
**Service:** `screen.service` (Type=simple, Restart=always, RestartSec=10)

### auto-update

Installs the cron job that drives the entire self-update mechanism. Without this role, the device stops receiving updates.

**Cron schedule:** Daily at 00:00
**Command:** `ansible-pull -U https://github.com/ORION-DEVELOPMENT-DIONE/update-manager.git -C main local.yml -i localhost, -o`
**Log:** `/var/log/ansible-pull.log`

**Tags:** `auto-update`

## Secrets Management

Secrets are stored in `group_vars/all/vault.yml` encrypted with Ansible Vault (AES-256).

**View secrets:**
```bash
ansible-vault view group_vars/all/vault.yml
```

**Edit secrets:**
```bash
ansible-vault edit group_vars/all/vault.yml
```

**Required vault variables:**
```yaml
vault_tailscale_oauth_client_id: "your-client-id"
vault_tailscale_oauth_client_secret: "tskey-client-..."
```

**Important:** Never store secrets in `vars.yml`. That file is committed to git unencrypted. Use `ansible-vault` for all sensitive values.

## Usage

### Manual Run (for testing)

```bash
# Full provisioning run
ansible-playbook local.yml -i localhost, --ask-vault-pass

# Dry run (check mode)
ansible-playbook local.yml -i localhost, --check --ask-vault-pass

# Run specific role only
ansible-playbook local.yml -i localhost, --tags screen-manager --ask-vault-pass

# Run without Tailscale (no vault needed)
ansible-playbook local.yml -i localhost, --tags "network,screen-manager,auto-update"
```

### Remote Run (via Tailscale SSH)

```bash
# SSH into device
ssh root@orion-<hostname>

# Check last update log
tail -50 /var/log/ansible-pull.log

# Trigger manual update
ansible-pull -U https://github.com/ORION-DEVELOPMENT-DIONE/update-manager.git \
  -C main local.yml -i localhost,

# Check screen-manager status
systemctl status screen.service
journalctl -u screen.service -f
```

### Adding a New Role

1. Create the role directory structure:
   ```bash
   mkdir -p roles/my-new-role/{tasks,files,templates,defaults,handlers,vars}
   ```

2. Write the main task file at `roles/my-new-role/tasks/main.yml`

3. Add the role to `local.yml`:
   ```yaml
   roles:
     # ... existing roles ...
     - role: my-new-role
       tags: my-new-role
   ```

4. Test locally before pushing:
   ```bash
   ansible-playbook local.yml -i localhost, --tags my-new-role --check
   ```

### Common Issues

**Ansible-pull fails with Jinja2 error:**
The playbook includes a pre_task that pins Jinja2 to v3.1.5 to work around an incompatibility with the system Ansible on Orange Pi. If this task itself fails, SSH in and run `pip3 install jinja2==3.1.5 --force-reinstall`.

**Screen service in restart loop:**
Check `journalctl -u screen.service -n 50` for the Python traceback. Common causes: missing font files, SPI permission error, MQTT broker unreachable.

**Tailscale not connecting:**
Check if the auth key is valid: `cat /etc/tailscale/key_metadata.json`. If expired, delete it and re-run the `tailscale-auth` role to generate a fresh key.

**Device unreachable after update:**
If a network-config change breaks connectivity, you'll need physical access. The safest approach is to test network role changes on a lab device before pushing to main.

## Compatibility

- **Target OS:** Ubuntu 22.04 (Jammy) on Orange Pi
- **Ansible:** 2.14+ (installed on device via pip3)
- **Python:** 3.10+ (system Python on Orange Pi)
- **Tailscale:** Latest stable from official APT repo
